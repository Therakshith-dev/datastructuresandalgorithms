Tree Data Structures

Ordered Abstract Data Types vs. Hash-Based Associative Containers

1. Formal Definitions

Binary Search Tree (BST)

A hierarchical data structure where each node contains a key (and associated value) and satisfies the binary search property: for any node, all keys in the left subtree are strictly less than the node's key, and all keys in the right subtree are strictly greater. This property enables logarithmic-time search operations when the tree remains balanced.

Invariant: ∀ node N: left_subtree.keys < N.key < right_subtree.keys

AVL Tree

A self-balancing BST where the heights of left and right subtrees differ by at most one (balance factor ∈ {-1, 0, 1}). Rotations maintain this invariant after mutations, guaranteeing height ≤ 1.44 log₂(n+2).

Red-Black Tree

A self-balancing BST with an additional color attribute per node (red or black) satisfying five invariants, ensuring that the longest path is no more than twice the shortest path. This yields height ≤ 2 log₂(n+1).

Hash Map (Hash Table)

An associative container that uses a hash function to map keys to array indices (buckets). Collisions are resolved via chaining or open addressing. Provides amortized O(1) insertion, deletion, and lookup but does not preserve order.

Hash Maps: O(1) access · Unordered · No range queries · No predecessor/successor
Trees: O(log n) access · Fully ordered · Logarithmic range queries · Predecessor/successor in O(log n)

4. Composite Data Storage

Tree nodes store key-value pairs where the value may be an arbitrarily complex record:

type TreeNode struct {
    Key   string      // Domain for comparisons
    Value UserRecord  // Associated complex data
    Left  *TreeNode
    Right *TreeNode
}

type UserRecord struct {
    Username    string
    Email       string
    Age         int
    Posts       []Post        // Slice (array) of posts
    Friends     []string      // Array of usernames
    Settings    map[string]bool
    CreatedAt   time.Time
}

Upon successful search by key, the entire composite value is accessible in O(log n) time, enabling:

  • Direct field access: user.Email, user.Age
  • Aggregate operations: len(user.Posts), user.Friends iteration
  • Mutations: user.Age++, user.Posts = append(user.Posts, newPost)
Key insight: The tree organizes by key only; the associated value can be any composite type, including arrays, maps, or nested structures — exactly like a hash map's value, but with ordered access.

5. Range Queries: The Tree Distinction

Hash maps cannot perform range queries without a full scan. Trees enable logarithmic-time range retrieval:

func (t *Tree) RangeQuery(low, high string) []Value {
    result := []Value{}
    t.rangeQueryRec(t.root, low, high, &result)
    return result
}

// Find all users with usernames between "alice" and "bob"
users := tree.RangeQuery("alice", "bob")

Real-World Applications

  • Database indexes: SELECT * FROM users WHERE age BETWEEN 25 AND 35
  • E-commerce: Products priced between $50 and $100
  • Analytics: Events timestamped within the last hour
  • Autocomplete: All dictionary words starting with "app"
Range queries on ordered trees: O(log n + k) · Hash maps: O(n) full scan

6. Mathematical Guarantees

AVL Tree Height Bound

Let N(h) be the minimum number of nodes in an AVL tree of height h:

N(h) = N(h-1) + N(h-2) + 1, with N(0) = 1, N(1) = 2

This recurrence yields N(h) ≈ φ^(h+2)/√5 - 1, where φ = (1+√5)/2 ≈ 1.618. Solving for h:

h ≤ 1.44 log₂(n+2) - 1.328

Red-Black Tree Height Bound

From the red-black invariants (no two reds adjacent, all paths have equal black height), the longest path ≤ 2 × shortest path, yielding:

h ≤ 2 log₂(n+1)

Hash Map Collision Resolution

With load factor α = n/m, expected chain length = α. Under simple uniform hashing, expected access time = O(1 + α).

7. Selection Criteria

Use Hash Map When:

  • Only exact-match lookups required
  • Order is irrelevant
  • Maximum throughput for insert/lookup is critical
  • Keys are uniformly distributed (UUIDs, random IDs)
  • Working set fits in memory

Examples: Caches, symbol tables, dictionaries, object pools

Use Balanced Tree When:

  • Range queries needed (min/max, intervals)
  • Sorted iteration required
  • Prefix or pattern searches
  • Predecessor/successor queries
  • Database indexing (B-tree/B+tree variants)

Examples: Database indexes, filesystems, autocomplete, event logs

8. Production Implementations

Database Systems

  • PostgreSQL, MySQL, Oracle: Use B+ trees for primary indexes
  • MongoDB: B-tree based indexes (WiredTiger storage engine)
  • SQLite: B-tree for tables and indexes

Standard Libraries

  • Java: TreeMap (RB tree), HashMap (hash table)
  • C++ STL: std::map (RB tree), std::unordered_map (hash table)
  • Go: map (hash table), external packages provide tree implementations
  • Python: dict (hash table), sortedcontainers (tree-based)

Operating Systems

  • Linux kernel: Red-black trees for virtual memory areas (VMAs), process scheduling
  • Windows NTFS: B+ trees for directory indexing
Fundamental Trade-off:
Hash Maps sacrifice order for constant-time operations.
Trees preserve order at logarithmic cost, enabling a richer query algebra.

9. Limitations

Hash Map Drawbacks

  • No ordering → cannot answer range queries
  • Poor worst-case performance (hash collisions)
  • Resizing overhead (rehashing)
  • Memory overhead for buckets and collision chains

Tree Drawbacks

  • O(log n) vs O(1) for exact lookups
  • Rotation overhead during writes
  • More complex implementation
  • Pointer overhead per node (left/right/parent/color/height)
In practice, hybrid systems use both: hash maps for cache, trees for indexes.
